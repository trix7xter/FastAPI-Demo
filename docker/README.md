# Docker: запуск проекта «на пальцах»

Этот файл объясняет, как поднять проект через Docker, **даже если ты никогда
раньше с этим не сталкивался**. Читай сверху вниз.

---

## 1. Что такое Docker и зачем он тут

Представь, что приложению для работы нужно сразу несколько программ:

- **база данных** (PostgreSQL) — где хранятся отели, пользователи, брони;
- **кэш и очередь задач** (Redis) — быстрое временное хранилище;
- **само веб-приложение** (FastAPI) — то, что отвечает на запросы в браузере;
- **фоновый обработчик** (Celery) — делает долгие задачи в фоне (например, шлёт письма);
- **дашборд** (Flower) — страница, где видно статус фоновых задач.

Поставить и настроить всё это руками на своём компьютере — долго и больно.
**Docker** упаковывает каждую такую программу в «контейнер» — изолированную
коробочку со всем необходимым внутри. А **docker compose** запускает все
коробочки разом и связывает их между собой.

---

## 2. Главные файлы и за что они отвечают

| Файл | Простыми словами |
|------|------------------|
| `Dockerfile` | Рецепт сборки образа нашего приложения: какой Python взять, какие библиотеки поставить, какой код скопировать. |
| `docker-compose.yml` | Список всех контейнеров (база, redis, приложение, celery, flower) и как они связаны. |
| `.dockerignore` | Что **не** класть в образ (например, локальное окружение `venv` и секреты). |
| `docker/app.sh` | Скрипт запуска веб-приложения: применить миграции БД и стартовать сервер. |
| `docker/celery.sh` | Скрипт запуска фонового обработчика или дашборда (зависит от аргумента). |
| `.env-non-dev` | Настройки (адреса, пароли) для запуска **внутри Docker**. |
| `.env` | Настройки для локального запуска **без** Docker (на твоём компьютере напрямую). |

## 3. Как запустить (3 шага)

> Нужно, чтобы был установлен **Docker Desktop** (на macOS/Windows) — он включает
> и `docker`, и `docker compose`.

1. Открой терминал в корне проекта (там, где лежит `docker-compose.yml`).
2. Собери и запусти всё одной командой:

   ```bash
   docker compose up --build
   ```

   - `--build` — пересобрать образ приложения (нужно при первом запуске и
     после изменений в коде или `requirements.txt`).
   - Docker сам поднимет базу, дождётся её готовности (healthcheck), затем
     запустит приложение, celery и flower **в правильном порядке**.

3. Когда в логах всё «устаканится», открой в браузере:

   | Что | Адрес |
   |-----|-------|
   | Приложение (API + Swagger-документация на `/docs`) | http://localhost:9000 |
   | Flower — мониторинг фоновых задач | http://localhost:5555 |
   | База данных PostgreSQL (если подключаться клиентом) | `localhost:5433` |

Остановить всё: нажми `Ctrl+C`, либо в другом окне терминала:

```bash
docker compose down
```

Данные базы при этом **сохранятся** (лежат в томе `postgres_data`).
Чтобы стереть и базу тоже: `docker compose down -v`.

---

## 4. Полезные команды

```bash
docker compose up -d          # запустить в фоне (терминал останется свободным)
docker compose logs -f        # смотреть логи всех сервисов
docker compose logs -f booking # логи только веб-приложения
docker compose ps             # какие контейнеры сейчас работают
docker compose restart booking # перезапустить только приложение
docker compose down           # остановить и удалить контейнеры (данные БД целы)
```

---

## 5. Важно про адреса (частая ошибка новичков)

Внутри Docker контейнеры обращаются друг к другу **по имени сервиса**, а не по
`localhost`. Поэтому в `.env-non-dev`:

- `DB_HOST=db`     — имя сервиса базы из `docker-compose.yml`;
- `REDIS_HOST=redis` — имя сервиса Redis.

Если поставить там `localhost`, контейнер приложения будет искать базу «внутри
самого себя» и не найдёт её. В обычном `.env` (запуск без Docker) наоборот —
там правильно `localhost`, потому что всё крутится на одной машине.
